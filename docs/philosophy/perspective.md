# Perspective

이 문서는 작성자가 소프트웨어를 왜 만들고, 어떤 태도로 다루는지 정리한다.

## Approach to Software

::: info 소프트웨어의 의미
작성자에게 소프트웨어는 목적을 달성하기 위한 도구이자, 문제를 정의하고 해결한 흔적이 남는 작업물이다.<br/>사용자가 필요로 하는 사항을 충족하거나, 소프트웨어 자체가 분명한 의미를 가질 때 가치가 생긴다.
:::

작성자는 구현 이전에 "왜 이 소프트웨어가 필요한가"를 먼저 생각한다.<br/>
이 질문에 답하지 못하면 기능의 우선순위도 흔들리고, 완성 이후의 방향도 쉽게 모호해진다고 보기 때문이다.

* **주체성** <Badge type="tip" text="Self-User" /> : 작성자는 스스로를 첫 사용자로 두는 감각을 중요하게 본다. 직접 사용할 의사가 없는 기능은 쉽게 과장되거나 방치되기 쉽다고 판단한다.
* **실용성** <Badge type="warning" text="Utility" /> : 보기 좋은 구조보다 실제 사용성과 의미를 우선한다. 구현이 화려해도 쓰이지 않으면 가치가 낮다고 본다.
* **명확성** <Badge type="info" text="Clarity" /> : 무엇을 만들고 왜 만드는지 설명할 수 있어야 한다. 목적을 설명하지 못하는 기능은 보류하거나 다시 정의한다.

## Ownership

작성자는 이해하지 못하는 구조 위에 작업량만 쌓이는 상태를 경계한다.<br/>
한 번의 구현보다, 이후에도 다시 열어 보고 수정할 수 있는 상태를 더 중요하게 본다.

* **통제 가능성** <Badge type="tip" text="Control" /> : 코드베이스의 핵심 흐름은 작성자 스스로 설명 가능해야 한다. 동작은 하지만 이유를 설명하지 못하는 상태를 신뢰하지 않는다.
* **지속 가능성** <Badge type="warning" text="Continuity" /> : 지금 빠르게 끝내는 선택이 다음 수정 비용을 과하게 만들면 좋은 선택으로 보지 않는다. 다시 손댈 수 있는 구조를 선호한다.

## Value

작성자는 소프트웨어의 가치가 결과 사용과 의미를 통해 증명된다고 본다.

* **요구의 충족**: 사용자가 필요로 하는 기능을 제공하거나, 겪고 있는 불편함을 해소하여 실질적인 만족을 주는 상태이다.
* **내재적 의미**: 소프트웨어로서 각자의 역할을 수행하며 전하고자 하는 방향이 분명한 상태이다.
* **개선의 여지**: 한 번 완성하고 끝나는 결과물보다, 다시 돌아와 다듬을 이유가 남는 소프트웨어를 더 가치 있게 본다.

> [!IMPORTANT] 가치 확립과 동기 부여
> 사용자의 만족을 통해 소프트웨어의 가치가 확립된다. 작성자는 이러한 결과로부터 개발의 보람과 지속적인 동기를 얻는다.
