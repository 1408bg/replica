# Decision Making

이 문서는 작성자가 기술과 구조를 선택할 때 반복해서 확인하는 기준을 정리한다.

## Dependency

작성자는 의존성 최소화를 지향한다. 이것은 도구 사용 자체를 거부하는 태도라기보다, 추가되는 비용과 통제 범위를 먼저 따지는 습관에 가깝다.

* **기준** <Badge type="tip" text="Zero-Dependency" /> : 기술 추가는 타당한 근거를 요구하는 행위이다. 어떠한 기술도 사용하지 않는 상태를 기본값으로 둔다.
* **도입 이유** <Badge type="info" text="Reason" /> : 성능, 생산성, 팀의 표준, 특정 기능의 필요처럼 도입 이유를 한 문장으로 설명할 수 있어야 한다.
* **교체 비용** <Badge type="warning" text="Cost" /> : 한 번 도입하면 구조 전반에 영향을 주는 기술일수록 더 엄격한 근거를 요구한다. 학습 비용과 교체 난이도도 함께 본다.
* **사례**: 리액트는 대중적인 협업 환경이나 React Flow, Recharts와 같은 특정 패키지의 기능이 필수적인 경우에 선택한다. 이외에는 HTML, CSS, Vanilla JS 또는 Vite와 Vanilla TS 환경을 우선적으로 검토한다.

## Sustainability and Speed

지속 가능한 개발은 유지보수 가능하고 확장성 있는 설계를 향해 가는 과정에서 자연스럽게 달성된다고 본다.<br/>
특정 원칙을 지켰다는 사실 자체를 목표로 두지는 않는다.

* **속도와 규모** <Badge type="warning" text="Scalability" /> : 구현 속도는 소프트웨어의 규모와 팀 구성에 따라 달라진다. 규모가 커질수록 프레임워크가 유도하는 구조를 활용해 유지보수성을 확보한다.
* **맥락 적합성** <Badge type="tip" text="Context" /> : 개인 프로젝트와 협업 프로젝트는 같은 방식으로 다루지 않는다. 혼자 빠르게 실험할 때와, 여러 사람이 오래 유지할 때의 기준을 구분한다.
* **유연성**: 일회성 코드나 소규모 소프트웨어에서는 빠르고 간결한 작성을 선택한다. 반대로 계속 확장될 작업은 초기에 구조를 더 신중하게 잡는다.

## Complexity and Problem Solving

작성자는 복잡도를 정면으로 밀어붙이기보다 쪼개서 다루는 편을 선호한다.<br/>
기능이 커질수록 책임과 흐름을 나누어 독립적인 단위로 관리한다. 중요한 것은 특정 패턴의 이름보다, 서로 다른 관심사가 한곳에 섞이지 않도록 경계를 세우는 일이다.

* **계층 분리** <Badge type="info" text="Layered" /> : 입출력, 도메인 로직, 저장소 접근처럼 역할이 다른 코드를 분리하여 각 계층의 책임을 명확히 한다. 필요하다면 익숙한 아키텍처 패턴을 참고한다.
* **절차 분리** <Badge type="tip" text="Procedural" /> : 거대한 작업은 한 번에 처리하지 않고 단계별로 분리한다. 필요하다면 반복 처리나 흐름 제어 도구를 예시로 참고할 수 있지만, 핵심은 작업 순서를 드러내는 것이다.
* **범위 재조정** <Badge type="warning" text="Scope" /> : 물리적 한계로 구현이 어렵다고 판단되면 무리하게 우회하지 않고 기획이나 범위를 다시 조정한다.
* **문제 해결**: 원인을 즉시 파악하기 어려운 상황에서는 AI나 타인에게 질문하여 작업을 위임하거나 도움을 받는다. 불완전한 대응보다 정확한 원인 규명을 우선한다.
